# ROADMAP — Pigeon Mobile

The development plan for the Pigeon mobile client. **Android ships first; iOS follows** by reusing the same shared Rust core. Read [`CLAUDE.md`](CLAUDE.md) for the rules and [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md) for the structure.

## Guiding principles

1. **Logic in Rust, UI in native.** The shared `pigeon-mobile-core` crate owns crypto, protocol, session, and sync. Each phase grows the core first, then the UI that drives it. (CLAUDE.md "Cardinal Rule".)
2. **Small, independently-tested, independently-committed slices** — the same discipline the homeserver was built with. A stage isn't done until its tests are green and its docs are in sync.
3. **The server's wire contract is fixed; we consume it.** No protocol invention. The reference CLI (`../pigeon/clients/cli`) is the canonical example for every flow.
4. **Offline-first from the start.** Reads from local store; the network reconciles in the background. The mobile network is hostile (flaky, slow, backgrounded).
5. **Don't pull work forward.** Phase M(N) refuses Phase M(N+1) features unless explicitly asked.

## Phase overview

| Phase | Theme | Gate |
|-------|-------|------|
| **M0** | Toolchain & skeleton | Rust→UniFFI→Kotlin round-trips on an emulator; CI green |
| **M1** | Identity: register/login/session | Real account login on a live homeserver |
| **M2** | Plaintext messaging: rooms, sync, timeline, send | Two devices chat in an unencrypted room |
| **M3** | E2EE (the headline) | Two devices exchange real MLS-encrypted messages |
| **M4** | Media, backup/restore, polish, push | Encrypted media + device recovery + notifications |
| **M5** | iOS | The iOS app reaches Android feature parity over the same core |
| **M6** | Hardening & release | Store-ready: perf, security pass, accessibility, CI/CD |

**Ordering rule:** M0 gates everything (no features until the toolchain round-trips). M1→M2 prove the core⇄UI boundary on real flows before M3 layers crypto on top. M5 (iOS) starts only once Android is feature-complete enough to mirror. M6 is continuous but formalized at the end.

## Phase M0 — Toolchain & skeleton

The whole project's risk is concentrated here: proving the Rust-core-via-UniFFI pipeline works before any feature depends on it. Don't skip ahead.

- [x] **M0.1 — `pigeon-mobile-core` crate.** ✅ `core/` is a `lib`+`cdylib`+`staticlib` crate depending on `pigeon-core` and `pigeon-crypto` by **path** (`../../pigeon/crates/*`). Builds and `cargo test`s in the dev container; a `self_test_crypto` test creates a real `pigeon-crypto` `Device` and asserts its 32-byte Ed25519 key, proving the whole `openmls` chain links and runs inside the core.
- [x] **M0.2 — UniFFI scaffolding.** ✅ UniFFI 0.28 in **proc-macro/library mode** (`uniffi::setup_scaffolding!()`, no `.udl`). Exposes `core_version()` + `self_test_crypto()` and a `CoreError` enum. Kotlin bindings generate from the built cdylib via the crate's own `uniffi-bindgen` bin (`coreVersion()` / `selfTestCrypto()` confirmed in the output).
- [x] **M0.3 — Android cross-compile.** ✅ `cargo-ndk` builds the `.so` for `arm64-v8a` + `x86_64` (verified: genuine aarch64 / x86-64 ELF). Toolchain pinned in `docker/Dockerfile`: NDK `26.3.11579264`, Rust targets `aarch64/x86_64/armv7-linux-android*`, JDK 17, Android platform/build-tools 34. **Host-arch note:** the NDK ships only x86_64 Linux toolchain binaries, so on Apple Silicon the Android lane runs in an **amd64 (Rosetta-emulated) container** (`android` service in compose); core dev stays on the native-arch `dev` service. min SDK 24 / target+compile SDK 34 (in `android/app/build.gradle.kts`).
- [x] **M0.4 — Hello-core Android app.** ✅ A bare Jetpack Compose app (`android/app`, `MainActivity.kt`) installs the M0.7 log sink, then calls `coreVersion()` + `selfTestCrypto()` through the generated UniFFI bindings and renders the results. `assembleDebug` produces a debug APK bundling both ABIs' `.so`. **Visual gate passed:** installed on an `arm64-v8a` API-36 emulator (host, Apple Silicon), the app renders `pigeon-mobile-core 0.1.0` (`core_version()`) and `pigeon-crypto Ed25519 key: 32 bytes` (`self_test_crypto()`) — values computed in Rust — and logcat shows the M0.7 log sink (`pigeon/pigeon_mobile_core: Hello-core: log sink installed`). No crash. The full Rust→cargo-ndk→UniFFI→Kotlin→Compose pipeline round-trips on a real device.
- [x] **M0.5 — Build glue.** ✅ `android/app/build.gradle.kts` wires two `Exec` tasks — `cargoNdkBuild` (cargo-ndk `.so` for `arm64-v8a` + `x86_64`) → `generateUniffiBindings` (UniFFI Kotlin codegen from the built cdylib) — into `preBuild`, so `./gradlew assembleDebug` rebuilds core, regenerates bindings, and packages the `.so` per ABI in one command.
- [x] **M0.6 — CI.** ✅ `.github/workflows/ci.yml`: a **core** lane (`cargo fmt --check`, `clippy -D warnings`, `test`) and an **android** lane (NDK + cargo-ndk + `./gradlew assembleDebug`), both checking out `pigeon` as a sibling for the path-deps. *(Placeholder `your-org/pigeon` repo ref to set before first run.)*
- [x] **M0.7 — FFI logging callback.** ✅ A host-installed `LogSink` callback interface + `set_log_sink`/`emit_test_log` in the core; the Hello-core app implements it as a Logcat forwarder. The core never assumes a platform logger.

**Exit criteria:** an emulator app displays a value computed in Rust; CI builds both lanes; the build is one command. No protocol code yet. **✅ MET:** the app displays Rust-computed values on an arm64 emulator (M0.4 above), `assembleDebug` is one command (M0.5), and CI covers both lanes (M0.6). **M0 is closed; M1 is active.**

> **Toolchain upgrade (2026-07-04).** The `android/` Gradle build was moved off the M0.3 pins to run on a **Linux/amd64** developer host: **Gradle 8.9 → 9.6.1**, **AGP 8.5.2 → 9.2.1**, **Kotlin 2.0.20 → 2.2.10**, **compile/target SDK 34 → 36**, host **NDK r28c**. AGP 9's **built-in Kotlin** meant dropping the `kotlin.android` plugin, pinning the Compose compiler plugin to AGP's bundled Kotlin (2.2.10), migrating `kotlinOptions` → `kotlin { compilerOptions }`, and feeding the generated UniFFI Kotlin via the **Variant API** (`variant.sources.kotlin`) instead of `sourceSets[...].java.srcDir`. The app (now the **M1 auth UI**, not the Hello-core screen) was **installed and run on an x86_64 API-36 emulator** — auth screen renders, the Rust `LogSink` callback reaches logcat, no crash. **⚠ Not yet done:** the Docker dev container / CI (`docker/Dockerfile`, M0.3/M0.6) still pins JDK 17 / SDK 34 / NDK 26 and must be bumped to satisfy AGP 9 before the android CI lane builds again.

## Phase M1 — Identity (register / login / session)

First real protocol surface. Keeps crypto out of scope — just HTTP + token lifecycle.

- [x] **M1.1 — HTTP client in the core.** ✅ `api.rs` (reqwest + rustls, versions/features mirroring the server): an `Api` client with homeserver base-URL config, bearer-token injection (`set_token`), and generic `get`/`post`/`put` verb helpers. `P_`-error bodies map to a typed `ApiError`/`ErrorCode` (all 12 documented codes + `Other` for forward-compat on a server version bump); matches the stable `errcode`, never the human text. Host-side unit tests cover the error mapping + URL normalization (no network). **Server discovery — settled, not deferred:** Pigeon has **no client-side discovery** in its wire contract. `.well-known/pigeon/server` is *federation* host discovery (`{"m.server": host}`, server-to-server, port 8448) — off-limits to a client that speaks only the Client–Server API. There is no `.well-known/pigeon/client`/`m.homeserver` equivalent; the reference CLI takes the homeserver URL directly. So the caller passes a full base URL, matching the CLI exactly — there is nothing correct to build here unless/until the server grows a client-discovery surface. *(Named endpoints + FFI exposure land in M1.2.)*
- [x] **M1.2 — register / login.** ✅ `session.rs`: async FFI `register(server, username, password)` / `login(server, user, password)` calling `/register` + `/login` (password flow) exactly as the reference CLI does, returning an opaque `PigeonClient` object. **The access token stays in the core** (inside the client's `Api`); the UI gets only the non-secret `Session` record (`user_id`, `device_id`, `server`) — Gotcha #1. `ApiError` maps to a typed `CoreError` (`Api{code}`/`Network`/`Protocol`) so the UI branches on the `P_` code. First async FFI surface (UniFFI tokio runtime; generates Kotlin `suspend` fns). **Validated** by mock-HTTP tests (`wiremock`, no Docker): request shapes, bearer injection, `P_`-error → typed code, and the full FFI path (`session::login`/`register` → `Session`/`CoreError`, incl. network-failure typing). **Validated against the real homeserver** by the oneshot e2e (`e2e/` crate — a separate, Docker-gated lane; see below): it spins the real `pigeon` server over TCP (real Postgres via testcontainers) and drives register/login/wrong-password/duplicate-register/restore through the core's FFI. ✅ passing. *(Token persistence = M1.3; logout FFI = M1.5.)*
- [x] **M1.3 — session persistence.** ✅ A host-provided `KeyStore` callback (`put`/`get`/`delete`, backed by the Android Keystore / iOS Keychain), installed once via `set_key_store`. On login/register the whole session blob (identity + **token**) is written through it — never the app DB in clear (Gotcha #1). `restore_session()` reloads it on launch and validates the token against `/account/whoami`, but is **offline-first**: a live token restores; a `P_UNKNOWN_TOKEN`/`P_MISSING_TOKEN` clears the dead session (→ `None`); a *network* failure restores optimistically (don't log out just because we're offline). Keystore failures map to `CoreError::Storage`. Host-tested (mock keystore + mock server): persist round-trip, restore-valid, revoked-clears, offline-keeps, empty→none. FFI generates `KeyStore` interface + `suspend restoreSession(): PigeonClient?`.
- [x] **M1.4 — auth UI.** ✅ Compose screens under `android/app/.../auth/`: `AuthScreen` (pick homeserver, register, login) with loading + per-error states, `HomeScreen` (signed-in identity), and a thin `AuthViewModel` driving the core's `register`/`login`/`restore_session` (offline-first restore on launch; the opaque `PigeonClient` handle kept in-core — Gotcha #1). `PigeonApp` installs the log sink + the `AndroidKeyStore` (EncryptedSharedPreferences → Android Keystore) at process start. `AuthError.authErrorMessage` maps every typed `CoreException` to a message, branching on the `P_` `ErrorCode` (never text). **Validated** by JVM unit tests (`AuthErrorTest`) + a green `assembleDebug`. *(Logout UI = M1.5.)* New Android deps (flagged): `lifecycle-viewmodel-compose`, `lifecycle-runtime-compose`, `kotlinx-coroutines-android`, `androidx.security:security-crypto`, `junit`.
- [x] **M1.5 — logout & token invalidation.** ✅ An async `PigeonClient.logout()` FFI: it revokes the token server-side via `POST /_pigeon/client/v1/logout` (best-effort, exactly as the reference CLI does — an unreachable server or an already-dead token still clears locally) and then wipes the persisted session from the keystore. A genuine keystore fault is surfaced as `CoreError::Storage` (the blob would otherwise silently restore next launch). The UI wires a Sign-out action into `HomeScreen` driving `AuthViewModel.logout()`: on success the `PigeonClient` handle is dropped and we return to the form; a keystore fault keeps the session live and shows the error for a retry. **Validated** by mock-HTTP tests in `core` (revoke-hit-with-bearer + local-clear; local-clear-even-when-revoke-fails + no-session-survives) and extended in the `e2e/` oneshot lane (logout against the real server clears locally *and* the token is proven dead server-side — a re-injected blob is rejected by the real `/account/whoami`). The `AuthErrorTest` already covers the `Storage` message.

**Exit criteria:** install the app, register or log into a real running homeserver, stay logged in across restarts, log out cleanly. End-to-end test against a oneshot homeserver. **Progress:** ✅ **M1 complete** — register/login/restore **and logout** are proven against a real oneshot homeserver by the `e2e/` crate, and the full auth+logout UI is built and building. Next: **M2 (sync + plaintext rooms)**.

## Phase M2 — Plaintext messaging (rooms, sync, timeline, send)

The core messaging loop, unencrypted. Proves the sync engine and the offline-first store before crypto complicates it.

- [x] **M2.1 — local store.** ✅ `core/src/store.rs` — offline-first SQLite via **`rusqlite` (bundled)** (dep decided & flagged over `sqlx`: lighter tree, clean `cargo-ndk` cross-compile, no build-time DB). Append-only `events` log keyed by content-hash `event_id` (idempotent re-sync — Gotcha #8); `room_state` folded last-writer-wins by DAG `depth`; single-row `sync_token` stored verbatim (Gotcha #5); membership = current `p.room.member`. Reads: `timeline` (depth-ordered, back-paginatable), `list_rooms` (folded name/topic/encryption), `membership`, `current_state`. `user_version` migrations, append-friendly schema. 13 host unit tests (idempotency, LWW fold incl. late-stale, ordering/pagination, room-scoping, malformed-abort-atomicity, reopen-persistence). **Internal module — not yet FFI-wired; the sync loop (M2.2) drives it.**
- [x] **M2.2 — the sync loop.** ✅ `core/src/sync.rs` — `PigeonClient.run_sync(observer)`: an endless async FFI long-polling `GET /sync` (`api.sync`, 30s poll / 100 limit), folding each batch into the store (`apply_sync` flattens `rooms.join[*].timeline.events` and applies in one idempotent transaction), advancing the opaque `next_batch` verbatim (Gotcha #5). Change stream via a `SyncObserver` callback (`on_change` when new events land, `on_status(connected)` for the offline indicator — coarse by design, Gotcha #7). **Cancellation (Gotcha #6):** the host runs it in a cancellable coroutine; UniFFI drops the future → the in-flight `/sync` is cancelled. Offline-first: transport errors back off (1→30s) and retry; only a fatal error (revoked token) returns `Err`. Store wired into `PigeonClient` via a startup `set_store_dir` (in-memory fallback when unset). Validated by unit tests (fold/idempotency/token-advance-on-empty) + mock-HTTP (`/sync` request shape; `run_sync` applies-notifies-then-cancels).
- [x] **M2.3 — room list + create/join.** ✅ Core (`core/src/rooms.rs`): `PigeonClient.list_rooms()` (sync store read → `Room` FFI record), `create_room(name, topic)` / `join_room(id)` (async; effects surface via sync). UI (`android/.../rooms/`): `RoomListScreen` + `RoomsViewModel` — the signed-in landing screen. `RoomListRoute` runs the sync loop in a client-keyed `LaunchedEffect` (leaving the screen / new session cancels it → Gotcha #6); its `SyncObserver` re-reads the store on change and drives an offline indicator. Create/join dialogs; sign-out moved here (old `HomeScreen` removed). Validated: mock-HTTP tests (create posts name/topic + omits unset; join path) + host `assembleDebug` (bindings regenerate, UI compiles) + JVM unit tests green.
- [x] **M2.4 — timeline + pagination.** ✅ Core (`rooms.rs`): `TimelineEvent` FFI record (message `body` **or** a core-rendered `system_text` for state/membership — no protocol parsing in native code, Gotcha #9; `cursor` = opaque load-older key), `PigeonClient.timeline(room, limit, before)` (paginated local store read), and `fetch_messages(room, limit)` (async backfill of recent history → persisted, deduped). **Server limit noted:** the messages endpoint has no older-than cursor, so it tops up recent history; deeper scroll-back pages through the store. UI (`rooms/Chat*`): a chat screen (message bubbles aligned by sender, centered system lines), opened by tapping a room; scroll-to-top pages older; a shared change signal refreshes the open timeline on the same sync events as the list. Validated: unit tests (event→body / system-line rendering, hidden types) + mock-HTTP (`fetch_messages` chunk persist/dedup + timeline read) + host `assembleDebug`.
- [x] **M2.5 — send (plaintext).** ✅ Core (`store.rs`/`rooms.rs`): offline-first send with **local echo**. `store.queue_send` writes a provisional echo (`events.local` = sending) + a `pending_sends` queue row (schema v2 migration); `PigeonClient.send_message` queues then flushes; `flush_pending` (re)transmits oldest-first — `resolve_send` promotes the echo to the server's event id (so the authoritative event from `/sync` dedups — no dup, no flicker), `fail_send` flags a rejected echo and dequeues it. The sync loop calls `flush_pending` each cycle (auto-retry when back online). `TimelineEvent` gained `pending`/`failed`. UI: a composer in the chat screen; own messages dim while sending and show "Sending…"/"Not sent". Validated: store unit tests (queue/resolve/promote/race-dedup/fail) + mock-HTTP (echo→confirm on ack; server-rejection→failed) + host `assembleDebug`.
- [x] **M2.6 — invites & membership.** ✅ Core (`rooms.rs`): `PigeonClient.invite(room, user)` (async → `POST /rooms/{id}/invite`); accepting an invite is `join_room(id)` (the server exposes no invite list in `/sync`, and there's no client leave/decline endpoint yet — both are server-side gaps, documented not stubbed). Membership changes already render in the timeline as core-pre-rendered system lines (`render_system`, M2.4 — Gotcha #9). UI (`rooms/Chat*`): an **Invite** action in the chat top bar opens a dialog (`InviteDialog`) that calls `ChatViewModel.invite`; the invite/join then surface in the timeline via the sync loop. Validated: mock-HTTP (`invite` posts `{user_id}` to the invite path with bearer) + the M2 exit-gate e2e (below) + `assembleDebug` + JVM unit tests. No new deps.

**Exit criteria:** two devices (or device + CLI) hold a conversation in an **unencrypted** room with live updates, pagination, offline send-and-retry. Integration test against a oneshot homeserver. **✅ MET** — the `e2e/` oneshot lane gained `two_clients_hold_a_plaintext_conversation`: Alice registers, creates a plaintext room, and invites Bob; Bob joins (membership); both run the real sync loop in the background; Alice's message reaches Bob and Bob's reply reaches Alice (round-trip over `/sync`), and Bob's join renders as a system line in Alice's timeline. Drives the same FFI the UI does, against the real `pigeon` server (Docker-gated; compiles in the container, runs on a Docker host). **M2 is complete; M3 (E2EE) is next.**

## Phase M3 — End-to-end encryption (the headline feature)

Wire `pigeon-crypto` (MLS) into the core and make encrypted rooms transparent in the UI. This is why the shared-Rust-core architecture exists. **Do not start until M1–M2 are solid** — debugging the core⇄UI boundary *and* MLS at once is a trap.

- [x] **M3.1 — device identity & key publishing.** ✅ Core (`core/src/e2ee.rs`): the `E2ee` engine — wraps `pigeon-crypto::Device` (the *only* crypto path; adds nothing of its own), mirroring the reference CLI's engine but persisting MLS state through the host `KeyStore` (Gotcha #1) rather than a file. `pigeon-crypto` has no pluggable storage backend, so every state-mutating op calls `export_storage()` and the engine persists `{pubkey, blob}` (base64) under `pigeon.mls.state.v1`; `restore(user_id, pubkey, blob)` reloads it on relaunch. `create`/`restore`/`clear`, `signature_public_key_b64`, `key_packages(n)`. Session wiring (`session.rs`): register/login mint a fresh identity and best-effort publish it (`publish_device_keys` → `api.upload_keys` → `POST /keys/upload`, `device_keys` + a pool of `KEY_PACKAGE_COUNT=5` base64 KeyPackages); `restore_session` restores the engine (or mints+publishes if absent); `logout` wipes the MLS state alongside the session blob. `CoreError::From<CryptoError>`; `api.rs` gained `upload_keys`/`query_keys`/`claim_keys`/`send_to_device`. Best-effort like the CLI — a key-upload failure logs a warning and leaves plaintext working, never blocking login. New dep (flagged): `base64 = "0.22"` (transport encoding only, not a crypto primitive; matches the server workspace). Validated: 5 host unit tests (persist↔restore round-trip, distinct KeyPackages survive persistence, no-state→None, corrupt-state→Storage error, clear) + the M1 session tests updated for the new MLS keystore entry; `cargo test`/`clippy -D warnings`/`fmt` green. No FFI surface change (the engine is internal). (Server 3F.7a/b.)
- [x] **M3.2 — key query / claim.** ✅ The HTTP verbs (`query_keys`/`claim_keys`) landed with the M3.1 api surface; M3.2 adds `core/src/keys.rs` — `claim_all_devices(api, user_id)`, the query→per-device-claim *sequencing* the reference CLI's `invite` performs (query the user's devices, then claim one KeyPackage from each; devices with none left are skipped; empty ⇒ the peer hasn't set up E2EE). Returns `ClaimedKeyPackage { device_id, key_package_b64 }` — the input to `add_member` (M3.4). Validated: 2 mock-HTTP unit tests (per-device claim with one device out of packages; no-published-keys → empty). (Server 3F.2/3F.3.)
- [x] **M3.3 — to-device + Welcome handling.** ✅ The `E2ee` engine gained the group ops (`has_group`/`create_group`/`add_member`/`join_from_welcome`, mirroring the CLI's `Engine`; group id = room id bytes). The sync loop (`sync.rs`) now processes the `to_device.events` block: for each `p.mls.welcome`, decode `content.welcome` and `join_from_welcome`, advancing the change signal so a newly-joined room refreshes. **Idempotent (Gotcha #8):** the Welcome carries its `room_id` out-of-band, so a re-delivered Welcome for a group we already hold is skipped; a bad/duplicate Welcome is logged and skipped, never wedging the loop (best-effort like the CLI). Validated: engine round-trip (create_group → add_member → join → shared membership) + garbage-Welcome negative test + sync unit tests (Welcome joins then dedups on replay; non-Welcome/malformed skipped). (Server 3F.4/3F.7c.)
- [x] **M3.4 — encrypted room creation + invite-with-Welcome.** ✅ Core (`rooms.rs`): `PigeonClient.create_encrypted_room(name, topic)` FFI — posts `createRoom {encryption:true}` (server materialises the `p.room.encryption` marker) then `create_group(room_id)` (group_id = room_id bytes); errors if E2EE is unavailable rather than making an unusable encrypted room. `invite` is now **transparent**: after the server invite, if we host the room's MLS group it runs the CLI's dance — `claim_all_devices` (M3.2) → `add_member` per device → ship each device its `Welcome` over `/sendToDevice` (`p.mls.welcome`, content `{welcome, room_id}`). Plaintext rooms skip it entirely. Validated: mock-HTTP (`create_encrypted_room` posts the flag) + a full invite-with-Welcome test with a **real** second-device KeyPackage (asserts the query/claim/sendToDevice request shapes *and* pulls the shipped Welcome back out of the recorded request to prove the invitee can actually join the group from it — real crypto, not just shapes) + `assembleDebug` (new FFI binding compiles). (Server 3F.6a/3F.7d.)
- [ ] **M3.5 — encrypted send/receive.** Encrypt to `p.room.encrypted`; decrypt inbound transparently and **cache the plaintext locally on first decrypt** (the ratchet advances and is persisted — Gotcha #3). UI shows encrypted rooms identically to plaintext, with an encryption indicator.
- [ ] **M3.6 — encrypted-room UX.** Lock indicators, "encryption enabled" event, graceful `[unable to decrypt]` fallback, device-list visibility.

**Exit criteria:** two devices exchange **real MLS-encrypted** messages through a homeserver that only ever sees ciphertext (assert the stored event has no plaintext — mirror the server's `e2ee_message` test). Negative tests: outsider can't decrypt; tampered ciphertext fails cleanly. **This satisfies the project's headline success criterion.**

## Phase M4 — Media, backup/restore, polish, push

Round out the client to "daily-driver" quality.

- [ ] **M4.1 — media upload/download.** `media.rs`: upload (`/_pigeon/media/v1/upload`, respect the size cap → handle 413), download by `pigeon://` URI, thumbnails. Image picker + viewer UI.
- [ ] **M4.2 — encrypted media.** Client-side encrypt before upload / decrypt after download — the server stores opaque ciphertext (no special server path; CLAUDE.md). Inline encrypted images in encrypted rooms.
- [ ] **M4.3 — encrypted device backup & restore.** Wrap `pigeon-crypto`'s recovery-key backup (server 3F.7e): `backup` produces a recovery key + stores the encrypted blob via the key-backup endpoints; `restore` on a fresh device recovers identity + groups. UI: show/save the recovery key, restore flow. Document the forward-secrecy caveat.
- [ ] **M4.4 — push notifications.** FCM integration; wake/sync on push; notify on new messages **without leaking plaintext** in the notification path until decrypted locally. (Confirm the homeserver's push contract first — may be out of this repo's scope.)
- [ ] **M4.5 — polish.** Read markers/receipts (if the server supports them), typing indicators (ditto), profile/display name + avatar, settings, theming, accessibility pass.

**Exit criteria:** send/receive encrypted media; lose and recover a device via the recovery key; receive a push and see a (locally-decrypted) notification. Each with tests.

## Phase M5 — iOS

Reuse the **same** core; add the second native UI. The payoff of the architecture.

- [ ] **M5.1 — core as `xcframework`.** Cross-compile the Rust core for `aarch64-apple-ios` (+ simulator targets); package via UniFFI's Swift bindings into an `xcframework`. CI lane for it.
- [ ] **M5.2 — Swift bindings smoke test.** A Hello-core SwiftUI app calls `core_version()` (mirror of M0.4).
- [ ] **M5.3 — iOS OS integration.** Keychain for secrets, APNs for push, native pickers/sharing, background-refresh-aware sync.
- [ ] **M5.4 — feature parity.** SwiftUI screens for M1–M4 flows, driven by the shared core. No new core logic should be needed; any that is, is a sign of a leaky boundary — fix it in the core for both platforms.

**Exit criteria:** the iOS app reaches Android feature parity with **no protocol/crypto code written in Swift** — only UI + OS glue.

## Phase M6 — Hardening & release

Continuous, but formalized for store readiness.

- [ ] **M6.1 — performance.** FFI-call batching audit (Gotcha #7), sync/timeline/decrypt benchmarks, binary-size trim (strip, per-ABI bundles), cold-start time.
- [ ] **M6.2 — security pass.** Confirm no plaintext/keys in logs/crash/analytics; keystore usage review; cert/TLS posture; dependency scan (`cargo audit`/`cargo deny` on the core, plus the Android/iOS dep trees); run the server repo's `/security-review` over the diff. An *external* audit is organizational, not a code deliverable.

## Cross-cutting success criteria (the honest framing)

- **"It's a real E2EE client"** — *validated* by M3's end-to-end ciphertext-only test, not asserted.
- **"Android + iOS share a core"** — *demonstrated* when M5 reaches parity with zero protocol code in Swift. If Swift needs protocol logic, the boundary leaked — fix the core.
- **"Production-ready"** — store-submittable build + security/accessibility passes (M6). An *external* security audit and real-world uptime are organizational milestones beyond code.
- **Protocol fidelity** — every flow matches the reference CLI's sequence of calls; the federation never sees plaintext. This is non-negotiable and gated per-phase, not deferred to M6.

## Out of scope (flag, don't half-build)

- **MLS membership churn** beyond the server's documented cap (member removal, concurrent/conflicting commits, multi-device epoch reconciliation) — inherits the server repo's 3F scope limit. The app targets small, add-mostly groups until the protocol grows.
- **Running federation on-device.** The app is a Client–Server API consumer only.
- **A web client.** Separate effort; not this repo.
- **Server-side push infrastructure.** This repo integrates against whatever the homeserver exposes; it doesn't build the push fan-out.
